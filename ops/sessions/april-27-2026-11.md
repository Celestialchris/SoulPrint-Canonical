## Why this workflow had to go

Three reasons, in priority order:

1. **It never deployed anything.** GitHub Pages was never enabled for the repo, so every run failed at the `actions/configure-pages@v4` step with `Error: Get Pages site failed. Please verify that the repository has Pages enabled and configured to build using GitHub Actions` followed by `HttpError: Not Found`. The workflow has produced exactly zero successful deploys in its lifetime.
2. **soulprint.dev is on Netlify, not GitHub Pages.** The site is served from Netlify (site slug `superb-pothos-0472e9`). Auto-deploy via the Netlify GitHub App webhook is the canonical deploy path, and `netlify.toml` at the repo root locks the publish directory to `landing` explicitly. There is no second deploy target that the workflow was protecting; it was a vestigial path from before the Netlify decision.
3. **Failing checks pollute the commit signal.** Every push to `landing/**` produced a red ❌ on the commit and on every subsequent landing PR. That conditioned reviewers to ignore failing checks on landing-touching PRs, which is the exact failure mode that erodes signal quality on a real test failure later. Removing the workflow restores the rule that a red ❌ means something is actually broken.

## Design decisions

Three alternatives considered and rejected during the cleanup brief:

1. **Keep the workflow and enable GitHub Pages in repo settings as a parallel deploy.** Rejected. Two parallel deploys serving the same content would create coherence debt and a real possibility of the two diverging silently. Netlify is canonical; one deploy path is the right number.
2. **Delete the entire `.github/workflows/` directory.** Rejected. The directory holds active, working CI: `tests.yml` (test suite), `release.yml` (release automation), `claude.yml` and `claude-code-review.yml` (Claude integration). A clean sweep would take all four down to remove one, which is the wrong tradeoff.
3. **Audit and refactor all five workflows in this branch.** Rejected. Branch scope discipline matters; sweeping CI/CD changes are their own branch. This branch deletes one file and adds one session log.
