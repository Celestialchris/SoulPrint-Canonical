## April 22, 2026 — CP1 security followup: _safe_next hardening (CodeQL #12-15)

**Branch:** fix/safe-next-url-validation

**What shipped:**
Hardened `_safe_next` to close CodeQL py/url-redirection alerts #12, #13, #14, #15. Added `from urllib.parse import urlparse` to the stdlib import block in `src/app/__init__.py`. Replaced the `startswith("/") and not startswith("//")` guard with a two-stage check: (1) reject immediately if any backslash is present; (2) parse with `urlparse` and reject if `netloc` or `scheme` is non-empty. Expanded `test_star_rejects_bad_next` in both `StarImportedRouteTest` and `StarMemoryRouteTest` from 3 cases to 7 cases, adding: `"https://evil.com/path"`, `"\\\\evil.com"`, `"/\\evil.com"`, `"javascript:alert(1)"`.

**The real bug:**
The original `startswith("/")` check had a bypass: `/\evil.com` passes because it starts with `/` but not `//`. Browsers normalize the backslash to `/` in the Location header, turning `/\evil.com` into `//evil.com`, which resolves as a protocol-relative URL to `evil.com`.

**Implementation deviation from prompt:**
The prompt's target code used `nxt.replace("\\", "")` before calling `urlparse`. This approach is semantically broken for the test cases the prompt itself specified: `"/\\evil.com"` strips to `"/evil.com"` (same-origin path, passes `urlparse` check) and `"\\\\evil.com"` strips to `"evil.com"` (bare hostname, no netloc or scheme, passes check). Both then contain `"evil.com"` in the Location header, failing `assertNotIn("evil.com", location)`. Switched to reject-on-backslash: any `next` value containing `\` is treated as invalid and falls back to the endpoint. No legitimate internal redirect URL uses a backslash.

**CodeQL idiom:**
`urlparse(nxt).netloc` and `urlparse(nxt).scheme` are recognized taint sanitizers in CodeQL's py/url-redirection model. The backslash pre-filter is a simple string containment check before the recognized sanitizer, not a replacement for it. This combination closes #12-15.

**Decisions:**
- Reject-on-backslash rather than strip-then-validate. Safer because no legitimate value contains a backslash; stripping an attacker-supplied value and then accepting the result obscures the intent and risks subtler bypasses on future edits.
- `urlparse` check handles all scheme-present and netloc-present cases (`//`, `http://`, `https://`, `javascript:`).
- The `""` (empty string) case: `if nxt and ...` short-circuits on falsy value, returning fallback. No separate check needed.

**Test suite:** 930 passing (unchanged count; 8 new subtests across two test methods included in the `24 subtests passed` count).
