# Session — April 25, 2026 — fix/continuity-route-log-before-abort

## Symptom

When `generate_continuity_packet` returned a result with a non-None error string — for example, because the stub provider returned malformed JSON — the continuity generate route called `abort(500)` and discarded the upstream failure message. PowerShell showed only `POST /intelligence/continuity/<id> HTTP/1.1 500 -`. No diagnostic context was available without attaching a debugger.

## Root cause

`abort(500)` raises `werkzeug.exceptions.InternalServerError` immediately. Any local variable that existed before the call — including `result.error` — is discarded. Werkzeug's WSGI access log records only the status line. The `intelligence_continuity_generate` route at `src/app/__init__.py:1827` was the only `abort(500)` call in the entire `src/` tree, and it had no preceding log statement.

## What shipped

**One log line** (`logger.error(...)`) added immediately before `abort(500)` inside `intelligence_continuity_generate`. Logs `conversation_id` and `result.error` at ERROR level using lazy `%s` formatting. No other lines in the function or file were touched.

**One route test** (`test_post_generate_logs_error_when_provider_returns_invalid_json`) added to `tests/test_continuity_route.py`. Patches `StubProvider.complete` to return `"not valid json"`, wraps the POST in `self.assertLogs("src.app", level="ERROR")`, asserts 500 status, and asserts the captured log contains both `conversation_id=<id>` and `"Failed to parse provider response as JSON"`.

**One learned doc** at `ops/learned/flask-abort-log-before-discard.md` capturing the rule: any route that reads a result-object error string before `abort(N)` must log the string first.

Test suite: 1043 passed, 0 failed (up from 1042 before this session).

## What's still open

The transcript-budgeting fix for long conversations is a separate concern: the continuity service currently sends the full transcript in a single provider call. For very long conversations this risks hitting token limits or producing a truncated JSON response. That failure path is covered by the same `Failed to parse provider response as JSON` branch now visible in logs, but the budget cap itself is a distinct prompt.
