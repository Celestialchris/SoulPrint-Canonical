## Architecture choices

- **Three pure functions in `scorer.py`, no I/O.** `compute_crap`, `derive_function_coverage`, `score_tree`. The helper `derive_function_coverage` is technically CLI logic (it bridges coverage's line model to radon's block model), but keeping it pure means it can be exercised in 0.06s of unit tests without recursive pytest invocation.
- **Subprocess for coverage, library for radon.** `coverage.py` is invoked as `python -m coverage run -m pytest tests/` followed by `python -m coverage json -o <tmp>` because running pytest inside an in-process `coverage.Coverage` context plays badly with subprocess-spawning test workers. `radon.complexity.cc_visit` is a pure Python call and is used as a library.
- **Functions with no executable lines emit 0% coverage in the report**, not 100%. They stay visible in the ranking instead of being silently dropped. The pure helper returns `None` to keep the policy decision in one place (the CLI's `_join_coverage`), not baked into the math.
- **Reports are timestamped, never overwritten.** `report-2026-04-29.json` and `.md` per same-day run; subsequent runs the same day get a `-N` suffix on both files. Both are gitignored; the `.gitkeep` is the only thing the directory contributes to the repo.
- **Top-20 split.** JSON carries the full ranked list (440 entries on the first local run); Markdown shows top-20 only, for human triage.

## Test shape

12 new tests in `tests/test_quality_scorer.py`, organized into three `unittest.TestCase` classes:

- `ComputeCrapTest` (5 tests): formula at three reference points (100% cov + low comp; 0% cov + high comp; 50% cov + medium comp), plus the `complexity == 0` synthetic guard, plus a `subTest`-driven check that `cov == 100` collapses to `comp` for any complexity.
- `DeriveFunctionCoverageTest` (3 tests): synthetic block matches spec (`lineno=10..20`, executed `{10,12,14}`, missing `{11,13,15,16,17,18,19,20}` → 3/11 × 100), no-executable-lines returns `None`, lines outside the span are ignored.
- `ScoreTreeTest` (4 tests): results ranked descending, joined correctly by `(file, function)` (different files with the same function name stay distinct), missing coverage data is treated as 0%, and results are `ScoreResult` dataclass instances.

CLI orchestration in `cli.py` is not unit-tested; it is verified by manual smoke run only (Step 9 of the Template H prompt).
