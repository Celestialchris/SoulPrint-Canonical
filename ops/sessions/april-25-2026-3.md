# Session — April 25, 2026 — fix/continuity-tighten-transcript-budget

## Evidence from previous run

A manual continuity run against conversation 85 produced two diagnostic signals:

1. `continuity transcript: dropped 121 oldest messages to fit budget (150000 chars)` — the pre-call budget cap from the previous PR was active and working.
2. `level=WARN source=runner.go:187 msg="truncating input prompt" limit=65536 prompt=75363 keep=4 new=65536` — Ollama still truncated the assembled prompt to its 65,536-token context. 150,000 chars at the actual tokenizer density of ~2 chars/token rendered to 75,363 tokens, exceeding the limit.

The original 4 chars/token English-prose estimate used when setting 150,000 was off by 2x for this content.

## What shipped

- `MAX_TRANSCRIPT_CHARS` lowered from `150_000` to `80_000` in `src/intelligence/continuity/service.py`.
- Comment updated to reflect measured density (~2 chars/token), the math (80K chars → ~40K input tokens + 16K response + ~500 overhead = ~56.5K total, leaving ~9K headroom in the 65,536-token context), and guidance to measure empirically rather than estimate.
- Calibration note appended to `ops/learned/transcript-budget-before-provider.md`.
- All tests pass without modification (budget tests reference `MAX_TRANSCRIPT_CHARS` by import, not by hardcoded value).

## What to verify manually

Retry continuity generation on conversation 85 in the running app. In PowerShell, confirm:

- No `truncating input prompt` line from Ollama. If this warning is absent, the budget change successfully eliminated prompt overflow.

## Still open

Two scenarios after manual verification:

1. **Ollama still truncates** — the actual density is even higher than 2 chars/token for this specific conversation. Lower `MAX_TRANSCRIPT_CHARS` further (try 60,000) and re-test.
2. **No truncation but JSON parse still fails** — the `Invalid \escape` is in the model's raw output, not a truncation artifact. Next PR adds a narrow backslash-escape sanitization pass in `_parse_provider_response` before calling `json.loads`, applied only when the initial parse raises `JSONDecodeError` and the text contains `\\` sequences.
