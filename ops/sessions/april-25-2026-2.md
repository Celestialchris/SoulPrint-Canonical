# Session — April 25, 2026 — fix/continuity-transcript-budget

## Symptom

Generating a continuity packet for a long conversation (e.g., conversation 85 with 278 messages) returned HTTP 500. The log showed `Failed to parse provider response as JSON: Expecting value: line 1 column 1 (char 0)`. The provider (Ollama) received a prompt exceeding its 64K context window and silently truncated the input from the front, dropping the `_SYSTEM_PROMPT` JSON-output contract. The model emitted empty output. The service raised `JSONDecodeError`.

## Root cause

`_build_transcript` in `src/intelligence/continuity/service.py` had no budget. For 278 messages it produced a transcript well over 200K chars. Ollama's `keep=4` truncation strategy retains only the first 4 tokens of the input and the last N tokens up to the context limit, which can discard the system prompt entirely. There was no error visible on the SoulPrint side until the JSON parse failed.

## What shipped

**`MAX_TRANSCRIPT_CHARS = 150_000`** — module-level constant with rationale comment (~37K tokens at ~4 chars/token, leaving headroom in a 64K-context provider).

**`_render_message_for_transcript(message) -> str`** — extracted render helper, used by both the budget calculation and the final rendering, so size estimates are exact.

**`_truncate_messages_to_budget(messages, max_chars) -> tuple[list, int]`** — drops oldest messages while preserving at least the most-recent one. If the most-recent message alone exceeds the budget, it is tail-truncated with a `[message truncated]` prefix using a lightweight `_TruncatedMsg` stand-in (no SQLAlchemy dependency).

**`_build_transcript` updated** — now calls the helpers. When messages are dropped, the returned transcript starts with `[N earlier messages truncated to fit context]`.

**Three tests** added to `TranscriptBuildingTest` in `tests/test_continuity_service.py`: under-budget pass-through, over-budget drop with marker, single-giant-message tail truncation.

**One learned doc** at `ops/learned/transcript-budget-before-provider.md`.

Test suite: 1046 passed, 0 failed (up from 1043 before this session).

## What's still open

Chunked map-reduce continuity for very long threads is deferred. The current approach drops oldest context, which is correct for most conversations but loses historical decisions in very long threads. A future approach would chunk the transcript, summarize each chunk separately, then merge the summaries. That is a separate feature, not a bug fix.
