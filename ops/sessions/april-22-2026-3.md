## April 22, 2026 — CodeQL #12-15: inline open-redirect sanitizer at call sites

**Branch:** fix/safe-next-inline-sanitizer

**What shipped:**
Replaced `return redirect(_safe_next("endpoint"))` at all 4 star/unstar routes with an inline 4-line sanitizer block. No logic change — the inline block is equivalent to the helper. `_safe_next` stays defined for one more pass before removal.

**Root cause — interprocedural blindspot:**
CodeQL's py/url-redirection taint tracker recognizes `urlparse(x).netloc` and `urlparse(x).scheme` as sanitizers only when they appear in the same function body as the `redirect()` sink. Factoring them into `_safe_next` broke recognition even though the logic was identical. Alerts #12-15 fired on all 4 routes as a result. The fix is to inline, not refactor. This is the same class of issue as the `relative_to(home)` CodeQL case — the idiom must be visible at the sink.

**Learned-doc update:**
Added `**Interprocedural blindspot**` subsection to `ops/learned/codeql-taint-vs-relative-to.md` under the existing open-redirect section.

**`_safe_next` status:**
Intentionally retained. It is a nested function inside `create_app()`, not module-level, so no pyflakes/ruff unused-function warning fires. Removal is a separate followup pass.

**Decisions:**
- Inline identical logic at each sink rather than restructuring (e.g., inlining `_safe_next` body into its definition scope). Minimal diff, no behavior change, closes CodeQL.
- Did not push. Branch local, ready for PR when security scan confirms alerts closed.
