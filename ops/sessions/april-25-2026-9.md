# Session: April 25 2026 - feat/attachment-markdown-export-markers

## Branch
`feat/attachment-markdown-export-markers`

## What shipped
Phase 4 (partial): markdown export markers for conversation-level and message-level attachments. No asset files are copied, no `.assets/` folders are created, no `manifest.json` is written in this branch.

## Changes
- `src/app/__init__.py`:
  - Added `_safe_marker_name(original_filename)` helper that produces filesystem-safe names without regex.
  - Extended `render_conversation_markdown(...)` with optional kwargs `conversation_assets` and `message_assets`. When present, emits a `## Attachments` table near the top (conversation-level) and per-message `#### Attachments` blocks (message-level).
  - Both export routes (`export_conversation_markdown` and `export_selected_conversations`) now load assets via `list_conversation_assets` / `list_message_assets` and pass them to the renderer.
- `tests/test_conversation_export.py`: new `ConversationExportAttachmentsTest` class (6 tests).
- `tests/test_multi_select_export.py`: new `_seed_conversations_with_assets` helper and `MultiSelectExportAttachmentsTest` class (3 tests).

## Decisions
- Conversation-level attachment link format: `[[Title.assets/filename.ext]]` (Obsidian wiki-link, no embed prefix). Full SHA256 in the table.
- Message-level link format: `[[Title.assets/msg-NNN-filename.ext]] · mime/type · sha256`. Sequence index zero-padded to 3 digits.
- `assets_stem` (used in links) is derived from the same title sanitization logic as the markdown filename, computed at the top of the function so both are consistent.
- No filesystem access in the renderer — attachment bytes and physical paths are not touched.
- Obsidian export and browser zip behavior are unchanged; only the markdown content may now contain attachment markers.
