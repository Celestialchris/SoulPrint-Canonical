# Session: April 25, 2026 — fix/continuity-json-mode

## What prompted this

After PR #165 lowered `MAX_TRANSCRIPT_CHARS` to 80,000 chars, Ollama no longer hit the context-window limit on conversation 85. But the continuity endpoint still returned 500. The logged error was:

```
Failed to parse provider response as JSON: Extra data: line 1 column 10 (char 9)
```

This is reproducible in a Python REPL: `json.loads('"summary": "test"')` raises exactly `Extra data: line 1 column 10 (char 9)`. The model (Gemma3:4B via Ollama) was emitting top-level key-value pairs without enclosing braces. The previous run also logged `dropped 197 oldest messages to fit budget (80000 chars)`, confirming the truncation fix from PR #165 was working; the JSON malformation was a separate, model-side issue.

## What shipped

Branch: `fix/continuity-json-mode`

Changes:
- `src/intelligence/provider.py`: added `*, response_format: dict | None = None` keyword-only parameter to all four `complete` signatures (`LLMProvider` Protocol, `StubProvider`, `AnthropicProvider`, `OpenAIProvider`). `StubProvider` and `AnthropicProvider` accept and ignore it. `OpenAIProvider.complete` builds a `kwargs` dict and conditionally adds `response_format` and `temperature=0` before calling `client.chat.completions.create(**kwargs)`.
- `src/intelligence/continuity/service.py`: the `generate_continuity_packet` call site now passes `response_format={"type": "json_object"}` to `provider.complete`. This switches Ollama to grammar-constrained sampling for the continuity extraction.
- `tests/test_intelligence_provider_complete.py`: 2 new tests in `OpenAIProviderCompleteTest` — `test_response_format_omitted_by_default` (asserts `response_format` and `temperature` are absent when not requested) and `test_response_format_forwarded_when_requested` (asserts both are present with correct values when requested).
- `tests/test_continuity_service.py`: 1 new test class `JsonModeTest` with `test_continuity_requests_json_mode` — captures kwargs from a recording provider and asserts `response_format={"type": "json_object"}` and `max_tokens=16384`.
- `ops/learned/transcript-budget-before-provider.md`: appended grammar-constrained output lesson.

Test count before: 1046. Test count after: 1049. All 1049 pass.

## What to verify manually

Retry conversation 85 via the continuity UI. Two possible outcomes:

1. **Clean packet** — summary, decisions, open_loops, entity_map all populated with substantive content. This means the model is capable enough for the task and JSON mode resolved the parsing failure.
2. **Syntactically valid but semantically empty response** — `json.loads` succeeds but fields are empty or generic. This means grammar-constrained sampling is working (no parse error) but Gemma3:4B does not have sufficient instruction-following capability for this structured extraction task. In this case the fix is correct, the model is the bottleneck.

If outcome 2: the code change stands, but consider upgrading to `gemma3:12b` or `gemma3:27b` via `SOULPRINT_LLM_MODEL` before concluding the continuity surface is unusable on local hardware.

## What's still open

- If JSON mode produces semantically empty output for conversation 85, model upgrade is the next action. No code changes needed for that path.
- If semantic output is good, the continuity surface is healthy for Ollama with models at the 12B+ tier.
- Anthropic JSON mode (structured output via the messages API beta) remains out of scope for this PR. It would follow the same `response_format` parameter pattern but require provider-specific handling inside `AnthropicProvider.complete`.
